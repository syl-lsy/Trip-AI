## assistant

记忆系统是**纯文件层、零外部依赖**的架构，核心由以下 7 部分组成：

---

### 1. 5 层文件架构 (`docs/memory/`)

| 层 | 文件 | 用途 |
|---|---|---|
| Hot | `ACTIVE-CONTEXT.md` | 当前进行中状态（≤50行） |
| Warm | `MEMORY.md` | 长期决策/偏好/功能索引（≤200行） |
| Daily | `daily/YYYY-MM-DD.md` | 每日事件日志 |
| Topic | `topics/{agent}.md` | 各 agent 特有知识 |
| Cold | `archives/` | 历史归档 |

### 2. `opencode.json` 自动加载

```json
"instructions": [
  "docs/memory/MEMORY.md",
  "docs/memory/ACTIVE-CONTEXT.md",
  "docs/memory/MANAGEMENT.md",
  "docs/memory/AUTO-MEMORY.md"
]
```

每次会话启动时 OpenCode 自动加载 4 个记忆文件到上下文。

### 3. AUTO-MEMORY.md（纯指令文件）

这是核心——**不是代码实现，而是 prompt 指令**。文件定义了：
- **会话启动流程**：读 ACTIVE → 读昨日 daily → 扫描 MEMORY → 读今日 daily
- **每轮回复后检查清单**：7 项判断（用户纠正/架构决策/Bug 根因等）
- **写入规则**：先读后写、去重、单行格式、同轮合并

利用 OpenCode 的 `instructions` 最后加载特性（Recency 效应）提升 agent 遵循度。

### 4. 双重嵌入——dev agent prompt 末尾追加

每个 dev prompt 末尾都有记忆维护指令（如 `backend-dev.txt:48-51`）：
```
记忆维护（必做）：
- 完成任务后写入 docs/memory/daily/YYYY-MM-DD.md
- 共享知识 → 更新 docs/memory/MEMORY.md
- 后端特有知识 → 更新 docs/memo

## Claude Code Memory vs 当前记忆系统完整对比

深入研究了 Claude Code 泄露源码中曝光的完整记忆架构后，以下是差距分析：

---

### Claude Code 的 4 层记忆架构

| 层 | 实现方式 | 特点 |
|---|---|---|
| **CLAUDE.md** | 手动编写的规则文件（项目级/个人级/组织级） | 静态、手动维护 |
| **Auto Memory** | **运行时系统级能力** — 内建于 agent loop，自动判断→写入 MEMORY.md 索引（≤200行）+ 详细文件 | 自动捕获、运行时执行、grep 关键词检索 |
| **Auto Dream** | 后台子 Agent 自动触发（24h无维护 + ≥5新会话）或手动 `/dream` | 自动压缩、解决矛盾、删过期条目 |
| **KAIROS** | 未发布的常驻守护进程（代码中发现但未在任何公开版本中） | 持续观察-思考-行动循环 |

### 当前项目记忆系统

| 层 | 实现方式 | 特点 |
|---|---|---|
| **AGENTS.md + agent-workflow.md** | 手动写规则，opencode.json instructions 自动加载 | 等价 CLAUDE.md |
| **AUTO-MEMORY.md** | **纯 prompt 指令** — 要求 LLM 每轮回复后执行检查清单 | 无运行时支撑，依赖 LLM 遵循度 |
| **5 层文件架构** | MEMORY.md 同时充当索引和内容存储（一体式） | Claude Code 是索引+文件分离 |
| **compact-memory 脚本** | 手动运行的 TypeScript 脚本 | 需要人工触发，非后台自动 |

---

### 核心差距

| 维度 | Claude Code Auto Memory | 当前项目 |
|------|------------------------|---------|
| **执行机制** | ✅ **运行时内建** — agent loop 中自动执行内存写入 | ❌ 纯 **prompt 指令** — 依赖 LLM 每轮后自觉执行 |
