# 童行AI（Trip with Kids）产品需求文档

## 1. Executive Summary

### 1.1 Problem Statement

带娃出行的家长在规划亲子旅游时面临三大痛点：

- **信息碎片化**：交通、住宿、景点、餐饮等需要从多个平台分别搜集，且难以判断各环节是否适合儿童（如免票政策、儿童餐、安全设施）。
- **缺乏针对性**：通用旅游规划工具不考虑孩子年龄、作息节奏（午休需求）、亲子友好度等因素，生成的行程往往"大人走得动，孩子跟不上"。
- **知识获取成本高**：亲子出行涉及大量专业知识（婴儿车登机政策、儿童用药携带规范、景点身高限制），分散在各论坛和攻略中，缺乏结构化、可检索的知识库。

### 1.2 Proposed Solution

童行AI 是一个 AI 驱动的亲子旅游一站式规划平台，核心解决思路：

| 痛点 | 解决方案 |
|------|---------|
| 信息碎片化 | AI Agent 自动整合交通、住宿、景点、餐饮、预算，生成完整行程 |
| 缺乏针对性 | 基于孩子年龄的亲子友好度评估 + 按节奏（轻松/紧凑）定制行程 |
| 知识获取成本高 | pgvector RAG 知识库 + 行程节点自动关联相关知识卡片 |

用户通过自然语言对话输入需求（"三亚5天4晚，2大1小，孩子5岁，预算1万以内"），AI Agent 自动生成完整行程方案，并支持自然语言修改、方案对比、知识库联动。

### 1.3 Success Criteria

| 指标 | 目标 |
|------|------|
| MVP 核心功能完成度 | 智能规划工作台（核心对话 → 行程生成 → 可视化）全链路可用 |
| 知识库覆盖 | ≥ 200 条结构化亲子出行知识，覆盖 6 大出行阶段 |
| 行程生成成功率 | AI 能在 ≤ 15 秒内完成一次完整行程规划（含流式输出） |
| 知识库 RAG 命中率 | 用户提问中 ≥ 80% 能召回 ≥ 1 条相关知识 |
| 编码测试覆盖率 | 分支 ≥ 80%，函数 ≥ 80%，行 ≥ 80%，语句 ≥ 80% |

## 2. User Personas

### 2.1 核心用户画像：亲子家庭家长

| 属性 | 描述 |
|------|------|
| **身份** | 80/90 后家长，孩子 0-12 岁 |
| **技术素养** | 中高，习惯使用手机 App 和生活服务类产品 |
| **核心诉求** | 省心：不想花大量时间做攻略；安心：确保行程对孩子友好；省钱：了解优惠与免票政策 |
| **典型场景** | "周末想带娃去周边玩两天" / "暑假想带孩子去三亚，但担心太累" / "第一次带孩子坐飞机，不知道需要准备什么" |
| **决策关注点** | 孩子适不适合（年龄/身高限制）、节奏紧不紧（有没有午休）、亲子友好度评分 |

### 2.2 次要用户画像：运营管理员

| 属性 | 描述 |
|------|------|
| **身份** | 平台运营 / 内容编辑 |
| **职责** | 维护亲子知识库（新增/编辑/下架文章） |
| **使用场景** | 审核内容质量、更新季节性知识、管理目的地数据 |

## 4. AI System Requirements

### 4.1 整体架构

```
用户输入 → IntentRouter（意图分类）
              ├─ plan → TravelPlanner
              │           ├─ Transport Agent（交通方案）
              │           ├─ Accommodation Agent（住宿推荐）
              │           ├─ Attraction Agent（景点安排）
              │           └─ Budget Agent（预算计算）
              ├─ modify → TravelModifier（自然语言修改行程）
              └─ qa → KnowledgeQA（pgvector RAG 检索）
```

### 4.2 IntentRouter 意图分类

| 属性 | 说明 |
|------|------|
| **职责** | 接收用户自然语言输入，分类为 `plan` / `modify` / `qa` 三种意图 |
| **输入** | 用户文本消息 + 当前行程上下文（若有） |
| **输出** | `{ intent: "plan" | "modify" | "qa", confidence: number, entities?: {...} }` |
| **实现方式** | LangChain LLM 调用 + few-shot prompt 分类 |
| **延迟要求** | < 500ms |

**意图识别规则**：

- **plan**：用户提出新目的地、新行程规划请求（如"我想去三亚5天"、"帮我规划一个亲子游"）
- **modify**：用户对已有行程提出修改（如"放慢节奏"、"换酒店"、"第三天加个景点"）
- **qa**：用户提问相关知识（如"儿童机票怎么买"、"带什么药"）

### 4.3 TravelPlanner（行程规划 Agent）

#### 4.3.1 子 Agent 编排

| 子 Agent | 职责 | 依赖 |
|---------|------|------|
| **Transport Agent** | 查询出发地 → 目的地的交通方案（高铁/航班），比较耗时和费用 | 无 |
| **Accommodation Agent** | 推荐适合年龄段的亲子酒店，标注儿童设施 | Transport Agent 确定到达时间 |
| **Attraction Agent** | 按天推荐景点，考虑儿童节奏、午休、免票政策 | Accommodation Agent 确定住宿位置 |
| **Budget Agent** | 汇总交通 + 住宿 + 餐饮 + 门票，计算总预算 | 前三者完成 |

#### 4.3.2 编排流程

```
Step 1: 同时查询 Transport Agent 和 Accommodation Agent（无依赖，并行）
Step 2: 等到 Transport + Accommodation 完成 → Attraction Agent
Step 3: 等到 Attraction 完成 → Budget Agent（汇总）
```

#### 4.3.3 Agent Prompt 核心要素

每个子 Agent 必须要求 LLM 输出结构化 JSON，格式：

```json
{
  "items": [
    {
      "type": "transport" | "accommodation" | "attraction" | "dining" | "rest",
      "day": 1,
      "time": "08:00 - 08:36",
      "title": "苏州站 → 上海虹桥站 · 高铁 G1234",
      "cost": 80,
      "childFriendly": true,
      "notes": ["儿童免票（6岁以下）"],
      "knowledgeRefs": ["7"]  // 关联知识库 ID 列表
    }
  ]
}
```

#### 4.3.4 外部数据来源约束

- **住宿/景点/交通数据**：使用已收录的内置数据 + 外部 API 查询结果（具体实现见非功能性需求）
- **预订链接**：仅提供平台搜索页 URL（如 `https://flights.ctrip.com/search/...`），不能指向具体结果页
- **费用展示**：标注为"预估"，实际价格以预订平台为准

### 4.4 TravelModifier（自然语言修改 Agent）

| 属性 | 说明 |
|------|------|
| **职责** | 接收用户的修改意图，对当前行程进行局部/全局调整 |
| **典型请求** | "放慢行程节奏"、"更换亲子酒店"、"总预算压缩到8000内"、"增加美食推荐"、"第三天上午改成动物园" |
| **实现方式** | 将当前行程 JSON + 用户修改请求发送给 LLM，LLM 返回修改后的 JSON |
| **约束** | 仅修改用户指定的部分，未指定部分保持不变；修改后需重新计算预算 |

**关键设计决策**：
- 每次修改触发一次完整的 LLM 调用（不做增量 patch），以确保行程整体一致性
- 修改后的行程自动触发 Budget Agent 重新计算预算

### 4.5 KnowledgeQA（RAG 知识问答）

| 属性 | 说明 |
|------|------|
| **知识库存储** | PostgreSQL + pgvector（1536 维嵌入，使用 text-embedding-ada-002 或同等级嵌入模型） |
| **检索方式** | 混合检索：语义相似度（pgvector cosine distance）+ 关键词 BM25 加权 |
| **chunk 策略** | 按章节分块（每篇知识文章按 `## 一级标题` 分割），每块约 300-500 tokens |
| **召回数量** | top_k = 5，按相似度排序 |
| **来源引用** | 所有回答必须标注来源知识文章标题，格式为"📖 [标题] · 来源：童行亲子知识库" |
| **输出格式** | 文本回答 + 可点击的知识卡片引用（含知识文章 ID） |

**RAG 流程**：

```
用户问："儿童机票怎么买？"
  → 意图识别：qa
  → 向量检索：cosine_similarity(question_embedding, knowledge_chunks)
  → top_k=5 召回
  → LLM 基于召回内容生成回答（附加引用来源）
  → 返回文本 + 知识卡片引用
```

**知识覆盖范围（初始 MVP）**：

| 分类 | 最少条数 |
|------|---------|
| 目的地指南（三亚/云南/成都/上海） | 各 ≥ 5 条 |
| 出行准备 | ≥ 8 条 |
| 交通出行（飞机/高铁/自驾） | ≥ 8 条 |
| 住宿攻略 | ≥ 6 条 |
| 景点游玩 | ≥ 6 条 |
| 餐饮美食 | ≥ 4 条 |
| 安全应急 | ≥ 6 条 |
| **总计** | ≥ 48 条 |

### 4.6 流式输出与进度骨架屏

#### 4.6.1 进度骨架屏

AI 生成过程中，右侧面板显示逐步进度指示：

| 步骤 | 展示文案 | 状态图标 |
|------|---------|---------|
| 1 | 已读取出行需求 | ✅ (完成后) |
| 2 | 已查询交通信息 | ✅ (完成后) |
| 3 | 正在比价航班方案… | 🔄 (当前) |
| 4 | 检索景点与酒店 | ⭕ (待办) |
| 5 | 生成每日行程 | ⭕ (待办) |

#### 4.6.2 流式输出要求

- AI 回复（包括文本和骨架屏状态更新）使用 Server-Sent Events (SSE) 流式推送
- 每个子 Agent 完成时推送状态更新到前端
- 最终行程 JSON 生成后，前端解析并渲染可视化行程卡片

### 4.7 LLM 缓存策略

| 缓存场景 | 缓存键 | TTL | 说明 |
|---------|--------|-----|------|
| 意图分类结果 | `intent:{用户消息哈希}` | 1 小时 | 相同问题不重复分类 |
| 知识库检索结果 | `qa:{问题哈希}:top5` | 24 小时 | 知识内容相对稳定 |
| 交通方案查询 | `transport:{出发地}:{目的地}` | 6 小时 | 班次时刻变化较慢 |
| 行程生成 | 不缓存 | — | 行程因人而异，缓存收益低 |

缓存存储于 Redis，预估每日减少 LLM API 调用约 40%。

### 4.8 PII 保护约束

| 约束 | 说明 |
|------|------|
| 禁止发送至 LLM | 手机号、真实姓名、详细家庭地址、身份证号 |
| 代称处理 | 用户消息中如出现手机号/姓名，在传给 LLM 前替换为代称（如"用户1"） |
| 数据存储 | 用户个人信息与行程数据分离存储，行程数据中不包含 PII |
| 日志脱敏 | 所有日志输出前过滤 PII 字段 |

### 4.9 LLM 幻觉防护

| 策略 | 说明 |
|------|------|
| 知识源约束 | KnowledgeQA 回答必须基于召回结果，若召回结果无法回答问题，明确告知"暂未收录相关信息" |
| 费用展示 | 所有费用标注为"预估"，提示"以实际预订平台为准" |
| 外部链接 | 仅提供平台搜索页 URL，不提供具体搜索结果链接 |
| 输出验证 | Budget Agent 输出的费用数字需校验合理性（如单人单日餐饮 ≤ 800 元） |

## 7. Risks & Roadmap

### 7.1 技术风险评估

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| **LLM 响应延迟** — 多 Agent 编排导致单次规划 > 20 秒 | 用户体验差 | 高 | 进度骨架屏 + 流式输出 + 子 Agent 并行执行（LangGraph 并行节点） |
| **RAG 召回质量低** — 用户问题无法匹配相关知识 | AI 回答不可靠，产生幻觉 | 中 | 混合检索（语义+关键词）+ 人工标注高价值知识 + LLM 阈值判断（低于阈值告知"未收录"） |
| **Token 成本失控** — 每次规划消耗大量 token | 运营成本高 | 高 | LLM 缓存（Redis）、prompt 长度控制、选择高性价比模型 |
| **行程 JSON 解析失败** — LLM 输出格式不稳定 | 前端渲染失败 | 中 | 输出校验 + 重试机制（最多重试 2 次）+ fallback 展示原始文本 |
| **数据安全** — 用户手机号等 PII 泄露 | 合规风险 | 低 | PII 过滤 + 日志脱敏 + 数据传输加密 |
